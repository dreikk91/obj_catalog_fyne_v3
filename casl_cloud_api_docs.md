# Документація API (CASL CLOUD)

Ця документація описує REST API веб-додатку CASL Cloud, що працює за адресою `http://10.32.1.221:50003/`.

## Статус реалізації в нашому коді (2026-03-29)

- Базова інтеграція реалізована в `pkg/data/casl_cloud_provider.go`.
- Розширене покриття всіх наразі підтверджених endpoint/command реалізовано в `pkg/data/casl_cloud_api_support.go`.
- Додано підтримку read-only команд (`read_mgr`, `read_connections`, `read_grd_room`, `read_events`, `get_firmware_list`, `read_from_basket`, `monitor`, `get_msg_translator_by_device_type`) та action-команд (`group_on_device`, `group_off_device`, `update_grd_object`, `grd_obj_pick`, `grd_obj_finish`).
- Автотести API-обгорток: `pkg/data/casl_cloud_api_support_test.go`.
- UI частина realtime (WS stream ingestion) поки не завершена, хоча `subscribe` endpoint вже підтримано.

## Архітектура API

API побудовано за концепцією **Command Pattern** (єдиної точки входу). Замість набору різних RESTful-шляхів (`/objects`, `/devices` тощо), майже всі запити після авторизації відправляються на єдиний маршрут `/command` методом `POST`. Тип необхідної дії чи об'єкта визначається параметром `"type"` всередині JSON-тіла запиту.

- **Base URL**: `http://10.32.1.221:50003`
- **Кодування / Формат даних**: `application/json`
- **Авторизація**: JWT-токен, що передається у кожному запиті до маршруту `/command`.

## 1. Авторизація (Login)

Ендпоінт, що відповідає за перевірку облікових даних і видачу JWT-токена та інформації про сесію.

- **URL**: `/login`
- **Метод**: `POST`
- **Тіло запиту**:
```json
{
  "email": "test@lot.lviv.ua",
  "pwd": "test123",
  "pult_id": "1",
  "captcha": null
}
```

- **Формат відповіді (успішна авторизація)**:
```json
{
  "status": "ok",                  // Статус відповіді
  "user_id": "483",                // Внутрішній ID користувача в системі
  "fio": "Тестовенко Користува Тестович", // ПІБ авторизованого користувача
  "token": "eyJhbGciOiJIUzI1...",  // JWT-токен для наступних запитів (ОБОВ'ЯЗКОВО)
  "ws_url": "ws://10.32.1.221:23322", // URL для підключення до WebSockets (підписки)
  "server_control": {
    "host": "10.32.1.221",         // IP-адреса головного сервера
    "port": 50001                  // Порт сервера керування
  }
}
```
> Примітка: Отриманий `token` додається в тіло (body) усіх наступних запитів до `/command`.

## 2. Універсальний маршрут (`/command`)

Для вичитки списків об'єктів, подій та людей використовується URL `/command` з методом `POST`.

### 2.1. Отримання списку об'єктів
Відповідає за вичитку карток об'єктів (наприклад, квартир, магазинів), що знаходяться під охороною.

- **Тіло запиту**:
```json
{
  "type": "read_grd_object",
  "skip": 0,
  "limit": 100000,
  "token": "ВАШ_JWT_ТОКЕН"
}
```
- **Формат відповіді**:
```json
[
  {
    "obj_id": "1001",              // Унікальний ідентифікатор об'єкта в БД (OBJUIN)
    "number": "54",                // Пультовий номер об'єкта (OBJN)
    "name": "Магазин 'Продукти'",  // Повна/коротка назва об'єкта
    "address": "вул. Шевченка, 1", // Повна адреса об'єкта
    "lat": 49.839683,              // Географічна широта (для карти)
    "long": 24.029717,             // Географічна довгота
    "description": "Вхід з двору", // Додатковий опис та примітки
    "device_id": ["456"],          // Масив ID підключених пристроїв (ППК)
    "in_charge": ["789", "790"],   // Масив ID користувачів, відповідальних за об'єкт
    "status": {
      "alarm": 0,                  // Стан тривоги: 0 - норма, 1 - тривога
      "guard": 1                   // Стан охорони: 0 - знято, 1 - під охороною, 2 - частково
    }
  }
]
```

### 2.2. Отримання стрічки подій (Tape)
Лог подій системи (постановка/зняття з охорони, тривоги, тестові сигнали).

- **Тіло запиту**:
```json
{
  "type": "get_general_tape_objects",
  "token": "ВАШ_JWT_ТОКЕН"
}
```
- **Формат відповіді**:
```json
[
  {
    "event_id": "845920",          // Унікальний номер події
    "time": "2023-10-25T14:30:00Z",// Час виникнення події 
    "obj_id": "1001",              // ID об'єкта, якого стосується подія
    "event_type": "alarm",         // Тип події (див. розділ "Константи подій" нижче)
    "reasonAlarm": "12",           // Внутрішній ID причини події (з довідника AlarmReasons)
    "zone": 4,                     // Номер зони (шлейфа), на якій виникла подія
    "user_id": "789",              // Хто зняв/поставив (для подій arm/disarm)
    "description": "Пожежа у зоні 4"// Деталізований текстовий опис (в деяких версіях API може формуватись на льоту)
  }
]
```

### 2.3. Інформація про пристрої (Devices / ППК)
Повертає список обладнання (ППК – прилад приймально-контрольний), зареєстрованого в системі.

- **Тіло запиту**:
```json
{
  "type": "read_device",
  "skip": 0,
  "limit": 100000,
  "token": "ВАШ_JWT_ТОКЕН"
}
```
- **Формат відповіді**:
```json
[
  {
    "device_id": "456",            // Унікальний ідентифікатор ППК
    "number": "54",                // Пультовий номер, що співпадає з об'єктом
    "name": "ППК Магазин",         // Назва пристрою
    "type": "TYPE_DEVICE_Dunay_4L",// Константа типу моделі (зчитується з read_dictionary)
    "sim1": "+380501234567",       // Номер першої SIM-карти пристрою
    "sim2": "",                    // Номер резервної SIM-карти
    "online": true,                // Чи перебуває пристрій на зв'язку зараз
    "lines": [                     // Налаштування шлейфів (зон) 
      { "id": 1, "name": "Вхідні двері", "type": "entry" },
      { "id": 2, "name": "Вікно зал", "type": "motion" }
    ]
  }
]
```

- **Тіло запиту**:
```json

}
```
- **Формат відповіді**:
```json
[
  {
    "user_id": "789",              // Унікальний ID користувача-відповідального
    "last_name": "Іваненко",       // Прізвище
    "first_name": "Іван",          // Ім'я
    "middle_name": "Іванович",     // По батькові

    "passwords": ["1234"]          // Коди для клавіатури об'єкта (опціонально)

]
```

### 2.5. Повний перелік додаткових команд (Read-only API)

Нижче наведено повний список унікальних команд на зчитування даних, ідентифікованих під час повного обходу всіх модулів системи:

| Команда (`type`) | Опис | Отримані дані |
| :--- | :--- | :--- |
| **`read_pult`** | Перелік пультів моніторингу | Назви пультів, їхні координати (`lat`, `lng`) та список прив'язаних користувачів. |
| **`read_mgr`** | Мобільні групи реагування (МГР) | Список груп МГР, їхні номери, телефони екіпажів та статус реєстрації в системі. |
| **`read_connections`** | Зв'язки (Об'єкт-Прилад-Людина) | Комплексна таблиця, що показує, яка людина відповідає за який пристрій на конкретному об'єкті. |
| **`read_events_by_id`** | Логи та журнал подій | Архівні події за період (`time_start` / `time_end`) для обраних об'єктів (використовується в звітах). |
| **`read_grd_room`** | Список приміщень об'єкта | Деталізація зон/приміщень всередині об'єкта, наявність RTSP-потоків для відеокамер. |
| **`get_disconnected_devices`**| Пристрої без зв'язку | Список усіх приладів (ППК), що на даний момент знаходяться в офлайні. |
| **`get_firmware_list`** | Довідник прошивок | Перелік доступних файлів прошивок (`.hex.enc`) для віддаленого оновлення обладнання. |
| **`read_from_basket`** | Кошик (видалені записи) | Перегляд видалених об'єктів, пристроїв та зв'язків перед їх остаточним знищенням системою. |
| **`get_statistic`** | Статистика по об'єктах/модулях| Агреговані дані для відображення статистичних показників (кількість тривог, взято/знято тощо). |
| **`monitor`** | Стан системних модулів | Моніторинг доступності зовнішніх сервісів, бази даних та внутрішніх компонентів ПЦС. |
| **`get_msg_translator_by_device_type`** | Метадані для перекладу подій | Повертає словники для розшифровки специфічних кодів подій для різних типів приладів (напр. MAKS_PRO). |

#### 2.6. Загальний журнал (General Journal)
Використовується для отримання повного потоку подій системи за визначений період.

- **Команда (`type`)**: `read_events`
- **Тіло запиту**:
```json
{
  "type": "read_events",
  "time_start": 1774735200000,   // Unix timestamp (ms) - початок
  "time_end": 1774817999999,     // Unix timestamp (ms) - кінець
  "time_request": 1774772513736, // Поточний час запиту
  "token": "ВАШ_JWT_ТОКЕН"
}
```
- **Особливість**: Запит зазвичай повертає всі події за вказаний період. Фільтрація за номером об'єкта чи типом події в веб-інтерфейсі часто відбувається на стороні клієнта після завантаження масиву.
- **Формат події у відповіді**:
```json
{
  "ppk_num": 1021,        // Номер приладу (ППК)
  "time": 1774771008000,  // Час події
  "code": 2092,           // Числовий код події
  "contact_id": "R402",   // Код ContactID (напр. R402 - взяття)
  "number": 1             // Номер зони або групи
}
```

#### 2.7. Журнал ППК та аудит (PPK Journal Mode)
Використовується для отримання детальної історії дій користувачів та подій по конкретних пристроях/об'єктах.

- **Команда (`type`)**: `read_events_by_id`
- **Тіло запиту**:
```json
{
  "type": "read_events_by_id",
  "isFullEventsInfo": true,
  "time_start": 1774735200307,
  "time_end": 1774818000307,
  "deviceNumbers": [1003],      // Масив номерів ППК для фільтрації
  "objIds": ["24"],             // Масив внутрішніх ID об'єктів
  "deviceIds": ["23"],          // Масив внутрішніх ID пристроїв
  "logReadJournal": false,
  "token": "ВАШ_JWT_ТОКЕН"
}
```
- **Особливість**: На відміну від загального журналу, цей запит підтримує серверну фільтрацію за конкретними ідентифікаторами. Використовується для побудови звітів по конкретному об'єкту.

#### 2.8. Каталог Звітів (Reports API)

Усі звіти генеруються через команду `get_statistic`. Нижче наведено детальний опис доступних звітів та структур їхніх відповідей.

- **Команда**: `get_statistic`
- **Параметри**: `name` (ідентифікатор звіту), `limit` (кількість записів).

| Назва звіту (`name`) | Опис | Ключові поля JSON |
| :--- | :--- | :--- |
| **`stats_events`** | **Історія подій** | `device_number`, `name` (об'єкт), `time` ("13:50:13, 29.03.2026"), `msg` (код події), `num` (шлейф). |
| **`stats_objects`** | **Реєстр об'єктів** | `number`, `name`, `address`, `description` (інструкція), `contract` (договір), `status`. |
| **`stats_activity`** | **Робота операторів** | `last_name`, `first_name`, `GRD_OBJ_PICK` (взяття), `GRD_OBJ_FINISH` (завершення), `DEVICE_BLOCK`. |
| **`stats_mgr_status`** | **Статус екіпажів МГР** | `name` (група), `status` ("в работе", "снята"). |
| **`stats_device_blocked`**| **Заблоковані пристрої** | `name`, `block_message` (причина блокування), `time_unblock`. |
| **`stats_users`** | **Список користувачів** | `email`, `phone_number`, `role` (ADMIN, MANAGER, TECHNICIAN, IN_CHARGE). |
| **`stats_devices_v2`** | **Обладнання та SIM** | `sim1`, `sim2`, `device_type` (напр. AX_PRO), `last` (час останнього зв'язку). |
| **`stats_devices_offline`**| **Offline пристрої** | `offline` (коли зник зв'язок), `address`, `sim1` (для зв'язку). |
| **`stats_connection`** | **Зони та Шлейфи** | `zone_number`, `zone_description` (напр. "СМК"), `line_type`, `group_number`. |

**Приклад JSON-відповіді (на прикладі `stats_events`):**
```json
{
  "status": "ok",
  "data": [
    {
      "device_number": 1,
      "name": "Ломбард УМКВ",
      "time": "13:06:47, 29.03.2026",
      "msg": "CHECK_CONN_FAIL",
      "num": 0
    }
  ]
}
```

#### 2.9. Кошик (Basket API)

У системі реалізовано механізм "Кошика" для тимчасового зберігання видалених записів (об'єктів, пристроїв, користувачів). Це дозволяє переглянути видалені дані або остаточно їх знищити.

- **Команда для читання**: `read_from_basket`
- **Тіло запиту**:
```json
{
  "type": "read_from_basket",
  "token": "ВАШ_JWT_ТОКЕН"
}
```
- **Формат відповіді**:
Система повертає об'єкт з масивами за типами даних. Кожен елемент має унікальний `basket_id`.
```json
{
  "status": "ok",
  "guardedObjects": [...],
  "devices": [...],
  "connections": [...],
  "users": [...],
  "mgrs": [...]
}
```

- **Команда для остаточного видалення**: `del_from_basket`
- **Тіло запиту**:
```json
{
  "type": "del_from_basket",
  "basket_ids": [14, 20], // Масив ID для видалення
  "token": "..."
}
```

- **Логіка відновлення (Restore)**:
У API немає окремої команди `restore`. Веб-клієнт виконує відновлення у два етапи:
1. Викликає стандартну команду створення (напр. `create_grd_object`) з даними, отриманими з кошика.
2. Після успішного створення викликає `del_from_basket` для цього `basket_id`.

## 3. Отримання текстового опису подій та робота з довідниками (Словниками)

Багато структур API передають внутрішні числові або текстові ідентифікатори (наприклад `reasonAlarm: "12"` або `type: "TYPE_DEVICE_Dunay_4L"`).
Для отримання розшифровки цих значень, у тому числі для отримання **текстового опису причин тривог (Alarm Reasons)**, типів об'єктів (Object Types) чи областей (Regions), використовується спеціальна команда для вичитки "довідника" (словника):

- **Тіло запиту**:
```json
{
  "type": "read_dictionary",
  "token": "ВАШ_JWT_ТОКЕН"
}
```
Цей запит повертає великий JSON-об'єкт із підкаталогами. Наприклад:
```json
{
  "alarm_reasons": {
    "1": "Несправність мережі 220В",
    "12": "Спрацювала зона X (Тривога)",
    "40": "Втрата зв'язку з приладом"
  },
  "device_types": {
    "TYPE_DEVICE_Dunay_4L": "Дунай-4L",
    "AX_PRO": "Hikvision AX PRO"
  },
  "roles": {
    "IN_CHARGE": "Відповідальна особа",
    "TECHNICIAN": "Технік",
    "OPERATOR": "Оператор ПЦС"
  }
}
```
Фронтенд-додаток завжди зіставляє значення з отриманого списку подій (наприклад, `get_general_tape_objects`) з цим довідником, щоб відобразити користувачу зрозумілий текст події замість простого коду (наприклад, перетворює `reasonAlarm: "12"` на `"Спрацювала зона X (Тривога)"`).

## 4. Розшифровка констант подій (`event_type`)

Усі події (Events), що відображаються у стрічці та прив'язуються до об'єктів/пристроїв, мають певний тип. 
Нижче наведена класифікація констант, які приходять в API у полі `event_type` (або вираховуються внутрішньо):

| Константа (Тип) | Опис значення | Категорія/Рівень |
| :--- | :--- | :--- |
| `fire` | **ПОЖЕЖА** — Спрацювання пожежної зони. | Критична (Тривога) |
| `fault` | **НЕСПРАВНІСТЬ** — Поломка датчика або обрив шлейфу. | Критична |
| `restore` | **ВІДНОВЛЕННЯ** — Відновлення роботи датчика/зони після тривоги чи несправності. | Інфо |
| `arm` | **ПОСТАНОВКА** — Об'єкт взято під охорону (постановка). | Нормальна |
| `disarm` | **ЗНЯТТЯ** — Об'єкт знято з охорони. | Нормальна |
| `test` | **ТЕСТ** — Регулярний тестовий сигнал від ППК. | Службова |
| `power_fail`| **ВТРАТА 220В** — Зникнення основного живлення на пристрої. | Попередження |
| `power_ok` | **ВІДНОВЛЕННЯ 220В** — Основне живлення відновлено. | Інфо |
| `batt_low` | **НИЗЬКИЙ ЗАРЯД АКБ** — Розряджається резервний акумулятор. | Попередження |
| `offline` | **ВТРАТА ЗВ'ЯЗКУ** — Пристрій не вийшов на зв'язок у встановлений час. | Критична |
| `online` | **НА ЗВ'ЯЗКУ** — Пристрій знову з'явився в мережі після збою. | Інфо |
| `system` | **СИСТЕМА** — Системні події або помилки бази даних. | Службова |

## 5. Оновлення в реальному часі (Real-time Підписки)

CASL Cloud активно використовує підписки для інформування фронтенду про нові події без перезавантаження сторінки.

- **URL**: `/subscribe`
- **Метод**: `POST`
- **Тіло запиту**:
```json
{
  "conn_id": "...", 
  "tag": "ppk_in",
  "token": "ВАШ_JWT_ТОКЕН"
}
```
*`conn_id` – це ідентифікатор WebSocket-з'єднання, отриманого після конекту до `ws_url` з процесу авторизації.*
Серед найцікавіших тегів: `tape` (події в стрічці) та `ppk_in` (зміна станів шлейфів ППК).

## 6. Механізми оновлення даних в реальному часі

Система використовує гібридну модель для підтримки актуальності даних:

### 6.1. WebSockets та Підписки
Після авторизації додаток відкриває WebSocket-з'єднання з адресою, вказаною в `ws_url`. Основний потік нових подій іде через це з'єднання.

- **Стрічка тривог (Tape)**: Оновлюється виключно через WebSockets після підписки на тег `tape`. При завантаженні робиться початковий запит `get_general_tape_objects`, а всі наступні події прилітають через сокет.
- **Загальний журнал**: Працює аналогічно. Початкова вичитка — `read_events`, подальше наповнення новими рядками — через WebSocket.
- **Стани об'єктів (Взято/Знято)**: Змінюються в інтерфейсі миттєво при отриманні відповідного повідомлення через WebSocket (тег `ppk_in`).

### 6.2. Початкова вичитка та Пагінація
Для журналів та списків об'єктів завжди використовується HTTP POST запит до `/command`.
- Якщо ви перемикаєтесь між вкладками або змінюєте фільтр дат, додаток робить новий запит (наприклад, `read_events` або `read_events_by_id`).
- Для великих списків використовуються параметри `skip` та `limit`.

### 6.3. Об'єктний журнал (Історія об'єкта)
При відкритті картки об'єкта додаток робить запит:
```json
{
  "type": "read_events_by_id",
  "objIds": ["ID_ОБ'ЄКТА"],
  "isFullEventsInfo": true,
  "token": "..."
}
```
Це дозволяє отримати не тільки тривоги, а й повний аудит (хто з операторів відкривав картку, хто взяв тривогу в роботу — події типу `GRD_OBJ_PICK`, `GRD_OBJ_FINISH`).

## 7. Протокол взаємодії та Специфікація команд

Всі операції після авторизації виконуються через HTTP POST запит на `/command`.

### 7.1. Загальний формат запиту (Request Envelope)
Кожен запит повинен бути у форматі JSON і містити наступні обов'язкові поля:

```json
{
  "type": "назва_команди",   // Визначає дію (напр. read_device, group_on_device)
  "token": "JWT_TOKEN",     // Отриманий при логіні токен
  ...                       // Інші параметри, специфічні для команди
}
```

### 7.2. Загальний формат відповіді (Response Envelope)
Сервер завжди повертає JSON.

**Успішна відповідь**:
```json
{
  "status": "ok",
  "data": ... // Масив об'єктів або один об'єкт (може бути відсутнім для команд-дій)
}
```

**Відповідь з помилкою**:
```json
{
  "status": "error",
  "message": "Опис помилки для користувача",
  "code": 401 // Код помилки (напр. 401 - токен прострочено)
}
```

### 7.3. Специфікація команд-дій (Action Commands)

На відміну від команд читання (`read_*`), ці команди змінюють стан системи або об'єктів.

| Команда (`type`) | Параметри | Опис дії |
| :--- | :--- | :--- |
| **`group_on_device`** | `device_number` (int), `group_number` (int) | **Постановка під охорону**. Надсилає команду на ППК для взяття конкретної групи. |
| **`group_off_device`** | `device_number` (int), `group_number` (int) | **Зняття з охорони**. Надсилає команду на ППК для зняття групи. |
| **`update_grd_object`** | `obj_id` та набір полів (`name`, `address`, `lat`, `long` тощо) | **Оновлення картки об'єкта**. Зазвичай відправляється весь об'єкт цілком. |
| **`grd_obj_pick`** | `obj_id` або `event_id` | **Взяття тривоги в роботу**. Фіксує в системі, який оператор почав обробку. |
| **`grd_obj_finish`**| `obj_id` або `event_id` | **Завершення обробки**. Закриває картку тривоги, переводить її в архів. |

## 9. Алгоритм формування текстового опису подій

У системі CASL Cloud опис події (наприклад, *"Відповідь на опитування - норма шлейфа № 1"*) **не приходить** у готовому вигляді з сервера у відповіді `read_events`. Бекенд надсилає лише набір кодів та числових ідентифікаторів, а текстовий опис формується **динамічно на стороні клієнта**.

### 9.1. Вхідні дані події (JSON)
Типовий об'єкт події з API:
```json
{
  "ppk_num": 1,          // Пультовий номер приладу
  "contact_id": "E390",  // Код протоколу ContactID
  "code": 61184,         // Внутрішній код події
  "number": 4,           // Номер зони (шлейфа) або групи
  "time": 1774779207905
}
```

### 9.2. Крок 1: Отримання шаблону перекладу
Для кожного типу обладнання (Ajax, Dunay, Maks PRO) існують свої словники. Додаток робить запит:
- **Команда**: `get_msg_translator_by_device_type`
- **Результат**: Мапа, де ключем є `contact_id` або `code`, а значенням — шаблон рядка.

**Приклади мапінгу:**
*   `E390` -> `"Не прийшло опитування за вказаний час"`
*   `R402` -> `"Взяття групи № {number}"`
*   `61184` + тип 'Відповідь на опитування' -> `"Відповідь на опитування - норма шлейфа № {number}"`

### 9.3. Крок 2: Збагачення даними про об'єкт
Щоб додати інформацію про зону (як у прикладі *"Опис: Тривожна кнопка..."*), фронтенд звертається до конфігурації приладу, отриманої раніше через `read_device`.

- У полі `lines` приладу шукається запис із `id`, що відповідає `number` з події.
- Текст із поля `name` (напр. *"СМК вхід"*) додається до фінального рядка.

### 9.4. Крок 3: Фінальна збірка рядка
Додаток підставляє змінні у шаблон і додає метадані про об'єкт. Результат:
`"2026-03-29 13:06:34 | Відповідь на опитування - норма шлейфа № 4 | Опис: Тривожна кнопка..."`

## 11. WebSocket та підписки в реальному часі

Система використовує WebSocket для доставки подій у реальному часі. Оновлення журналів та стрічки тривог відбувається через один канал зв'язку.

### 11.1. Встановлення підписки (Subscribe)
Після підключення до `ws_url` клієнт має підписатися на потрібні потоки даних за допомогою HTTP POST запиту.

- **URL**: `/subscribe` (метод `POST`)
- **Тіло запиту**:
```json
{
  "conn_id": "ваш_ідентифікатор_ws", 
  "tag": "ppk_in",
  "token": "ВАШ_JWT_ТОКЕН"
}
```

- **Основні теги (`tag`)**:
    *   `ppk_in` — Вхідні події (тривоги, норми, зміна станів). Це основний потік для Журналів та Стрічки.
    *   `ppk_out` — Відповіді приладів на надіслані команди.
    *   `user_action` — Аудит дій інших операторів.
    *   `notif` — Системні сповіщення.

### 11.2. Стрічка тривог (Alarm Tape)
Специфічного тегу для "тільки тривог" не існує. 
1. Клієнт підписується на загальний тег `ppk_in`.
2. Клієнт фільтрує вхідні повідомлення на основі довідника `get_msg_translator_by_device_type`.
3. Якщо код події в трансляторі має прапорець `isAlarm: 1`, подія відображається у Стрічці тривог.

### 11.3. Приклад вхідного повідомлення (Message Data)
```json
{
  "ppk_num": 1,           // Номер приладу
  "time": 1774772911327,  // Час (Unix ms)
  "code": 16145,          // Внутрішній код події
  "type": "ppk_event",    // Тип повідомлення
  "number": 4,            // Номер зони (шлейфа)
  "contact_id": "E130"    // Код протоколу ContactID
}
```

## 13. Системні події та дії оператора (User Actions)

Окрім подій від заліза (ППК), система фіксує дії персоналу ПЦС. Ці події передаються через WebSocket за допомогою тегу **`user_action`**.

### 13.1. Основні команди керування тривогами
Всі дії оператора з карткою тривоги відправляються на `/command` з типом `grd_object_action`.

#### **Взяття в роботу (GRD_OBJ_PICK)**
Закріплює об'єкт за оператором.
```json
{
  "type": "grd_object_action",
  "action": "GRD_OBJ_PICK",
  "obj_id": "ID_ОБ'ЄКТА",
  "token": "..."
}
```

#### **Завершення обробки (GRD_OBJ_FINISH)**
Закриває картку тривоги з вказанням причини та примітки.
```json
{
  "type": "grd_object_action",
  "action": "GRD_OBJ_FINISH",
  "obj_id": "ID_ОБ'ЄКТА",
  "cause": "CAUSES_FALSE_ALARM", // Код зі словника causes
  "note": "Клієнт забув зняти з охорони",
  "token": "..."
}
```

### 13.2. Формат WebSocket-повідомлення для дії оператора
Коли один оператор здійснює дію, всі інші отримують повідомлення для оновлення інтерфейсу (наприклад, щоб тривога зникла із загальної стрічки).

**Типове повідомлення (`tag: user_action`):**
```json
{
  "type": "user_action",
  "dict_name": "GRD_OBJ_PICK", // або GRD_OBJ_FINISH
  "user_id": 483,              // Хто виконав дію
  "time": 1774790159852,
  "ppk_num": 1,
  "obj_id": "2"
}
```

### 13.3. Інші системні команди
*   **Відключення приладу**: `type: "change_disconnected_state"`. Використовується для тимчасового ігнорування сигналів від несправного ППК.
*   **Редагування пристрою**: `type: "update_device"`. Зміна примітки, таймауту або моделі.

## 14. Ієрархія та джерела словників подій

Розшифровка подій у системі CASL Cloud — це багатоступеневий процес, що поєднує дані з API та локальні стандарти.

### 14.1. Три рівні словників

| Рівень | Команда (`type`) | Що містить |
| :--- | :--- | :--- |
| **1. Системний** | `read_dictionary` | Загальні константи: причини тривог (`alarm_reasons`), назви моделей приладів (`device_types`), ролі користувачів. |
| **2. Апаратний** | `get_msg_translator_by_device_type`| Мапінг кодів (`code`) та ContactID (`contact_id`) для конкретних типів заліза (Ajax, Maks, Dunay). Містить шаблони повідомлень. |
| **3. Стандартний**| *Вбудовано в клієнт* | Резервна таблиця стандартних кодів **Ademco ContactID** (напр. E130 — проникнення, E110 — пожежа). |

### 14.2. Алгоритм формування опису події

Для створення читабельного рядка (*напр. "норма шлейфа № 2, Опис: СМК..."*) Go-клієнт має виконати такі кроки:

1.  **Отримання шаблону**: 
    - Використати `code` або `contact_id` з події для пошуку в трансляторі приладу (`get_msg_translator_by_device_type`).
    - Команда: `{"type": "get_msg_translator_by_device_type", "typeDevice": "MAKS_PRO"}`.
    - Результат: рядок виду `"норма шлейфа № {number}"`.

2.  **Підстановка параметрів**:
    - Вставити значення поля `number` (з події) у шаблон.
    - Результат: `"норма шлейфа № 2"`.

3.  **Збагачення даними шлейфа**:
    - Знайти назву шлейфа в конфігурації приладу (`read_device` -> поле `lines`).
    - Якщо `id` шлейфа = `number` події, взяти його `name` (напр. `"СМК двері"`).
    - Результат: `"норма шлейфа № 2, Опис: СМК двері"`.

### 14.3. Технічні деталі транслятора
У відповіді `get_msg_translator_by_device_type` для кожного коду окрім тексту зазвичай приходять прапорці:
*   `isAlarm`: 1 — подія критична, має потрапити в Стрічку тривог.
*   `alarmColor`: Код кольору для підсвічування рядка в журналі.
*   `priority`: Пріоритет обробки.

## 15. Технічна специфікація WebSocket-клієнта (Frontend Logic)

Цей розділ описує внутрішню логіку роботи з WebSocket, видобуту безпосередньо з вихідного коду веб-клієнта. Вона є критичною для стабільної роботи та синхронізації даних.

### 15.1. Життєвий цикл з'єднання та ініціалізація

Для коректного старту сесії клієнт має виконати наступні кроки:

1.  **Handshake**: Встановити WebSocket-з'єднання з `ws_url` (напр. `ws://10.32.1.221:23322`).
2.  **Auth (get_id)**: Одразу після `onopen` надіслати повідомлення:
    ```json
    { "type": "get_id", "user_id": "ВАШ_USER_ID" }
    ```
3.  **Session ID**: Сервер відповість повідомленням:
    ```json
    { "type": "conn_id", "id": "УНІКАЛЬНИЙ_ID_З'ЄДНАННЯ" }
    ```
4.  **Subscribe**: Після отримання `conn_id` клієнт має підписатися на ВСІ необхідні теги через HTTP POST `/subscribe`.

### 15.2. Повний перелік тегів для підписки

Веб-клієнт підписується на 13 тегів одночасно:

- `ppk_in` — Основні події приладів.
- `user_action` — Дії операторів (взяття, завершення).
- `ppk_service` — Сервісні повідомлення ППК (напр. прошивка).
- `notif` — Наднаступні системні сповіщення (вікна prompt).
- `system_event` — Події оновлення ПЗ сервера.
- `system_action` — Системні дії (звіти, доступ).
- `ping` — Keep-alive потік від сервера.
- `m3_in` — Стан груп швидкого реагування (МГР).
- `system_info` — Статистика буфера подій.
- `storage_change` — Повідомлення про зміну/видалення об'єктів у БД.
- `mob_user_action` — Дії мобільних користувачів.
- `chat_action` — Повідомлення чату.
- `rtsp_action` — Керування відеопотоками.

### 15.3. Механізми буферизації та стабільності

Фронтенд не відображає події миттєво, а використовує **буфери**, щоб уникнути "фризів" інтерфейсу при великому потоці:

- **PPK Buffer (1000мс)**: Групує вхідні події `ppk_in`.
- **User Action Buffer (1500мс)**: Обробляє дії операторів.
- **Lenta Buffer (2000мс)**: Специфічний буфер для "Стрічки" (Tape).
- **Time Filter (`clearWastedMsg`)**: Клієнт автоматично відфільтровує повідомлення, час яких (`time`) відрізняється від серверного більше ніж на **180 секунд** (120с затримка сервера + 60с допустима різниця годинників).

### 15.4. Синхронізація стану (`storage_change`)

Це найважливіший тег для підтримки актуальності бази даних у пам'яті (кешу).
Коли інший користувач змінює або видаляє об'єкт, сервер шле:
```json
{
  "type": "storage_change",
  "data": {
    "action": "DELETE_GUARDED_OBJECT_SUCCESS", // або UPDATE_...
    "info": "ID_ОБ'ЄКТА"
  }
}
```
**Дія клієнта**: Видалити або оновити відповідний об'єкт у локальному списку (Redux/State) без перечитування всього списку через HTTP.

### 15.5. Keep-alive та Реконект

- **Ping Timeout**: Клієнт очікує повідомлення від сервера кожні 10 секунд (`serverPeriod`). Якщо протягом 12 секунд (`timeForFlying + serverPeriod`) нічого не прийшло — з'єднання вважається розірваним.
- **Стратегія реконекту**: Таймер на 15 секунд для першої спроби, перехід на 30 секунд при повторних невдачах. Клієнт зупиняє всі буфери перед реконектом.

### 15.6. Рольова фільтрація сповіщень (`notif`)
Повідомлення типу `notif` (модулі `surgard`, `mozg`) відображаються користувачу лише якщо його роль вища за "Оператора" або це критичні помилки зв'язку сервера (напр. `SYSEV_ERR_SILENCE`).

## 16. Алгоритм низькорівневої розшифровки кодів (Protocol Decoding)

У системі CASL Cloud числове значення `code` (тип `int16`), яке приходить у `ppk_event`, є упакованим значенням, що потребує декодування за правилами конкретного протоколу.

### 16.1. Розподіл на байти (Split Bytes)
Для отримання значущих частин коду його потрібно розділити на два 8-бітні байти:
*   **b1 (High Byte)**: `code >> 8`
*   **b2 (Low Byte)**: `code & 0x00FF`

### 16.2. Порядок вибору транслятора
Декодування відбувається за ієрархією пріоритетів:

1.  **System Translator**: Якщо комбінація `(b1, b2)` знайдена в системному словнику, це повідомлення від внутрішніх модулів (напр. `0x00 0x60` -> `PPK_CONN_OK`, зв'язок відновлено).
2.  **Protocol Translator**: Якщо системного збігу немає, вибирається транслятор на основі моделі ППК (SIA, Dozor, VBD4, D128).
3.  **Sur-Gard Fallback**: Якщо для конкретного протоколу код не знайдено, використовується стандартний Sur-Gard мапінг (ContactID).

### 16.3. Розрахунок реального номера (`num`)
У багатьох протоколах значення `b2` не є кінцевим номером зони чи користувача. Використовується зміщення `codeDiff`:
**`num = b2 + codeDiff`**
*   *Наприклад*, якщо `b1 = 0x0A` (ID Господаря), а `codeDiff = 17`, то `b2 = 1` перетворюється на `num = 18`.

### 16.4. Приклади системних кодів (`systemDictionary`)

| b1 (Hex) | b2 (Hex) | Константа | Опис |
| :--- | :--- | :--- | :--- |
| `0x00` | `0x60` | `PPK_CONN_OK` | Зв'язок із приладом відновлено |
| `0x00` | `0xB3` | `BAN_TIME` | Заборона дій (тайм-аут) |
| `0x01` | `0x61` | `PPK_NO_CONN` | Втрата зв'язку з ППК |
| `0x00` | `0x67` | `SABOTAGE` | Саботаж (схоже на SIA) |

## 17. Класифікація обладнання та Протоколів

Щоб знати, який транслятор застосовувати, необхідно зіставити поле `type` з команди `read_device` із внутрішньою моделлю протоколу.

### 17.1. Мапінг типів на моделі (`getModelFromDeviceType`)

| Значення `type` у `read_device` | Модель (`PpkModel`) | Протокол |
| :--- | :--- | :--- |
| `TYPE_DEVICE_Dunay_4L` / `8L` | `Model4L` / `8L` | **Rcom** |
| `TYPE_DEVICE_Ajax` / `Lun` | `Ajax` / `Lun` | **Sur-Gard** |
| `TYPE_DEVICE_Ajax_SIA` | `SIA` | **SIA** |
| `TYPE_DEVICE_Dunay_4_3` / `VBD4_ECOM` | `VBD4` | **Ecom** |
| `TYPE_DEVICE_Dozor_4` / `8` | `Dozor` | **Ecom** |
| `TYPE_DEVICE_Dunay_16_32` / `8_32` | `D128` | **Ecom** |

### 17.2. Фільтрація не-тривожних повідомлень
Метод `isNoAlarmMessage` дозволяє автоматично відсіювати технічні події, які не повинні викликати тривожне вікно в оператора.
**Ключові не-тривожні константи**:
- `ACC_OK`, `OK_220`, `LINE_NORM`, `ZONE_NORM` (відновлення).
- `GROUP_ON`, `GROUP_OFF` (взяття/зняття).
- `RESTART`, `CHECK_CONN`, `PING` (сервісні).
- `ID_HOZ` (інфо про користувача).

### 17.3. Специфічні групи подій
*   **Line Alarms**: `LINE_BRK` (обрив), `LINE_KZ` (коротке замикання), `LINE_BAD` (несправність шлейфа).
*   **ContactID Protocol**: Для пристроїв `Ajax` та `Lun` розшифровка зазвичай базується на стандарті ContactID, якщо немає специфічного транслятора.
