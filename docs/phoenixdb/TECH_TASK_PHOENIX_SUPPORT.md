# Технічне завдання: впровадження підтримки БД Phoenix

## 1. Мета

Реалізувати в застосунку `obj_catalog_fyne_v3` підтримку БД Phoenix (`MS SQL Server`, інстанс `localhost\PHOENIX4`, база `Pult4DB`) як окремого джерела даних, яке може працювати як самостійно, так і одночасно з іншими БД та API-джерелами в мультиджерельній конфігурації застосунку.

Підтримка має забезпечити:

- одночасну підтримку трьох джерел даних: `MOST/міст`, `Phoenix`, `CASL API`;
- роботу основних екранів застосунку на даних Phoenix;
- коректне підключення Phoenix через налаштування;
- можливість одночасної роботи Phoenix з іншими БД та API;
- збереження поточної архітектури `application -> backend/contracts -> data`;
- мінімальні зміни в UI;
- для Phoenix admin-функціонал має бути окремим від Firebird/MOST-логіки;
- для Phoenix має бути окрема вкладка в налаштуваннях;
- можливість поетапного впровадження: спочатку read-only сценарії, потім адміністративні операції.

## 2. Підстава для робіт

### 2.1. Що є зараз у проєкті

Проєкт зараз жорстко орієнтований на Firebird:

- підключення відкривається драйвером `firebirdsql` у `pkg/database/connect.go`;
- DSN формується у Firebird-форматі в `pkg/config/db_config.go`;
- фабрика бекенда створює єдиний DB-провайдер через `backend.NewDBProvider(...)`;
- поточні SQL-запити працюють зі схемою MOST/Firebird (`OBJECTS_INFO`, `OBJECTS_STATE`, `OBJECTS_LA`, `EVLOG`, `MESSLIST`, `PERSONAL`, `ZONES`, `ACTALARMS`, `OBJTYPES`, `OBJREGS`, `SBS`, `TBL_TESTCONTROL`, `TRPMSG` та ін.);
- адмінський функціонал реалізований напряму через Firebird-таблиці й транзакції в `pkg/data/db_admin_provider*.go`.

### 2.2. Що відомо з документації Phoenix

За матеріалами в `docs/phoenixdb`:

- Phoenix працює на `MS SQL Server`;
- інстанс БД: `localhost\PHOENIX4`;
- основна база: `Pult4DB`;
- логін: `sa`;
- пароль: порожній рядок `''`;
- ключові сутності для об’єктів: `Panel`, `Groups`, `Company`, `MPhone`, `Zones`, `Responsibles`, `ResponsiblesList`, `ResponsibleTel`, `ResponsibleEmail`;
- важливі view: `vwRealPanel`, `vwArchives`, `vwMPhoneDeviceChannels`, `Code`, `TypeCode`;
- ідентифікатор об’єкта в Phoenix: `Panel.Panel_id`;
- один об’єкт може мати кілька груп (`Groups.Group_`);
- для багатьох прикладів у документації “основна” група об’єкта фактично береться як `group_ = 0`.

## 3. Межі робіт

### 3.1. У scope

- зафіксувати продуктову вимогу: застосунок повинен одночасно підтримувати `MOST/міст`, `Phoenix` та `CASL API`;
- додати новий режим бекенда `phoenix`;
- реалізувати окремий провайдер даних для Phoenix;
- зафіксувати архітектурне правило: Phoenix admin не залежить від мосту/MOST;
- реалізувати роботу Phoenix паралельно з іншими БД та API-джерелами;
- реалізувати окрему конфігурацію підключення до `MS SQL Server`;
- додати окрему вкладку налаштувань Phoenix;
- забезпечити роботу головних read-only сценаріїв:
  - список об’єктів;
  - картка об’єкта;
  - зони;
  - відповідальні;
  - журнал подій;
  - панель активних тривог;
  - технічна/канальна інформація по об’єкту;
- додати базові тести для мапінгу та маршрутизації нового провайдера;
- зберегти сумісність з Firebird/MOST.

### 3.2. Поза scope першої черги

- повне перенесення всіх адмінських сценаріїв Firebird 1-в-1 без додаткового аналізу Phoenix;
- автоматична міграція даних між Firebird і Phoenix;
- зміна UI-концепції;
- зміна CASL Cloud інтеграції.

## 5. Цільова архітектура

### 5.1. Загальний принцип

UI не повинен знати, з якою БД він працює. Вибір джерела даних має відбуватись на рівні конфігурації та фабрики провайдерів.

### 5.2. Архітектурне рішення

Потрібно запровадити:

- новий режим `BackendModePhoenix`;
- новий SQL-провайдер `PhoenixDataProvider`;
- новий шар запитів `pkg/phoenixdb` або `pkg/data/phoenix_*` для SQL Server;
- абстракцію підключення, яка вміє ініціалізувати різні драйвери/DSN;
- окремі мапери Phoenix -> `models.Object`, `models.Event`, `models.Alarm`, `models.Zone`, `models.Contact`.

Phoenix має вбудовуватись у вже наявну multi-source модель застосунку, а не замінювати її.

Це означає:

- застосунок повинен підтримувати одночасну роботу щонайменше трьох джерел: `MOST/міст`, `Phoenix`, `CASL API`;
- Phoenix повинен підтримуватись як одне з джерел у мультисистемній конфігурації;
- Phoenix може працювати паралельно з Firebird/MOST, CASL Cloud та іншими API-джерелами;
- маршрутизація об’єктів/тривог/подій повинна бути сумісною з поточним підходом `CombinedProvider` / `MultiSourceProvider`;
- архітектура не повинна примушувати користувача обирати тільки одне джерело даних на весь застосунок, якщо сценарій передбачає паралельне використання кількох систем.

Окреме правило:

- Phoenix admin-операції, якщо вони будуть реалізовані, повинні працювати напряму з Phoenix-схемою (`Panel`, `Groups`, `MPhone`, `Zones`, `Responsibles` тощо), без залежності від мосту, MOST-таблиць, Firebird-дзеркал або проміжного “БД/МІСТ” шару.

### 5.3. Рекомендована структура

- `pkg/config/db_config.go`
  - розширити конфігурацію режимом `phoenix`;
- `pkg/database/connect.go`
  - винести ініціалізацію драйвера в універсальний механізм;
- `pkg/backend/provider_factory.go`
  - додати фабрику Phoenix-провайдера;
- `pkg/data/combined_provider.go`
  - розширити конфігурацію джерел, маршрутизацію та правила ownership для Phoenix;
- `pkg/data/phoenix_provider.go`
  - read-only реалізація `contracts.DataProvider`;
- `pkg/data/phoenix_admin_provider.go`
  - admin-реалізація, якщо буде підтверджений scope другої черги;
- `pkg/data/phoenix_queries.go`
  - SQL-запити до Phoenix;
- `pkg/data/phoenix_models.go`
  - проміжні DTO для `sqlx`.

## 6. Функціональний мапінг Firebird -> Phoenix

### 6.0. Уніфікована модель груп для Phoenix і CASL

Оскільки застосунок повинен одночасно підтримувати `MOST/міст`, `Phoenix` і `CASL API`, для `Phoenix` та `CASL` потрібно зафіксувати спільну доменну модель роботи з групами.

Базове правило:

- універсальною оперативною сутністю для UI і backend має бути не `приміщення`, а `група об'єкта`;
- `приміщення` в `CASL` не є окремою головною сутністю навігації, а виступає контекстом групи;
- саме група повинна мати власний статус, власну назву для відображення, власний набір зон і власний набір відповідальних у вже зведеному вигляді.

#### Цільова уніфікована ієрархія

- `Object`
- `ObjectGroup`
- `Zone`
- `Responsible`

#### Інтерпретація для Phoenix

У `Phoenix`:

- `ObjectGroup` = реальний підоб’єкт `Panel + Group_`;
- кожна група має власну назву;
- кожна група має власний статус;
- кожна група має власні зони;
- кожна група має власних відповідальних.

#### Інтерпретація для CASL

В оригінальному `CASL`:

- об’єкт має групи;
- кожна група має власний статус;
- кожна зона прив’язана до певної групи;
- якщо зона не має явної прив’язки, вона належить першій групі за замовчуванням;
- групи прив’язані до приміщень;
- приміщення мають власні назви;
- відповідальні прив’язані до приміщень;
- фактично кожна група для UI повинна мати власну назву, власний статус, власні зони та власний набір відповідальних, отриманий через прив’язане приміщення.

#### Нормалізоване правило для backend

Для backend потрібно звести обидві системи до єдиної моделі:

- у `Phoenix` відповідальні належать групі напряму;
- у `CASL` відповідальні належать приміщенню, але на рівні застосунку мають резолвитись як відповідальні конкретної групи через `PremiseID`;
- у `CASL` назва групи для UI повинна зберігатись окремо від назви приміщення;
- у `CASL` назва приміщення повинна зберігатись як додатковий атрибут групи;
- у `CASL` кожна зона повинна мати явний `GroupID`, а якщо джерело його не повертає, бекенд повинен віднести таку зону до першої групи об’єкта.

#### Рекомендована структура доменної сутності

Пропонується додати уніфіковану модель `ObjectGroup` з такими полями:

- `GroupID`
- `ObjectID`
- `Source`
- `GroupNo`
- `Name`
- `Status`
- `PremiseID` optional
- `PremiseName` optional
- `Zones`
- `Responsibles`

#### Вимоги до UI

UI повинен працювати так:

- список зліва залишається на рівні об’єктів;
- всередині об’єкта групи не перетворюються на окремий другий список навігації;
- вкладка `Статус` повинна показувати компактний список усіх груп об’єкта;
- вкладки `Зони` і `Відповідальні` повинні показувати дані, згруповані по групах;
- вкладка `Події` на першій черзі може залишатись object-level, але модель подій має бути готовою до group-aware розширення;
- для `CASL` біля назви групи додатково можна показувати назву приміщення;
- `приміщення` в `CASL` не повинно вводитись як окремий рівень навігації, якщо це ускладнює уніфікацію з `Phoenix`.

Узгоджене рішення для першої реалізації:

- список об’єктів залишається object-level для всіх джерел;
- групи показуються всередині правої панелі об’єкта;
- для `MOST/міст`, де фактично є одна група, UI повинен залишитись максимально близьким до поточного;
- для `Phoenix` і `CASL` групування повинно проявлятись у вкладках деталей, а не в лівому списку об’єктів.

#### Наслідки для контрактів

Поточна object-centric модель контрактів недостатня для повної підтримки `Phoenix` і `CASL` у груповому режимі.

Потрібно передбачити нові group-aware контракти, наприклад:

- `ListObjectGroups(objectID string)`
- `GetGroupZones(groupID string)`
- `GetGroupResponsibles(groupID string)`
- `GetGroupEvents(groupID string)`
- `GetGroupStatus(groupID string)`

#### Наслідки для маршрутизації ідентифікаторів

Щоб уникнути конфліктів між `OBJN`, `Panel_id`, `CASL object id`, `group_` та внутрішніми груповими ключами, ідентифікатори груп потрібно робити namespaced.

Рекомендовані формати:

- `phoenix:panel=L00028:group=0`
- `casl:object=123:group=2`

#### Висновок

Для цього проєкту слід вважати, що:

- уніфікація `Phoenix` і `CASL` повинна будуватись навколо сутності `Group`;
- `Premise` у `CASL` є контекстом групи, а не універсальною заміною Phoenix-підоб’єкта;
- усі read-only і майбутні admin-сценарії для `Phoenix` і `CASL` слід проєктувати з урахуванням `Object -> Group -> Zones/Responsibles/Events`.

### 6.0.1. Технічна реалізація групової моделі для CASL і Phoenix

Нижче описано, як саме ця модель повинна збиратись технічно: які джерела даних використовуються, які проміжні структури потрібні і як результат компонується у backend.

#### Цільова backend-модель

Backend повинен формувати дві окремі проєкції:

- `ObjectAggregate` - агрегат рівня об’єкта;
- `ObjectGroupAggregate` - агрегат рівня групи.

`ObjectAggregate` повинен містити:

- базову інформацію про об’єкт;
- список груп;
- агрегований статус об’єкта;
- службові source-specific ключі.

`ObjectGroupAggregate` повинен містити:

- `GroupID`;
- `ObjectID`;
- `Source`;
- номер групи;
- назву групи;
- статус групи;
- назву приміщення, якщо вона є;
- список зон групи;
- список відповідальних групи;
- групові події;
- source metadata для навігації назад до джерела.

#### Рекомендований внутрішній пайплайн

Для обох джерел має бути однакова схема композиції:

1. Завантажити сирі записи об’єкта.
2. Побудувати root object.
3. Побудувати список сирих group-кандидатів.
4. Нормалізувати group-кандидати в `ObjectGroupAggregate`.
5. Прив’язати зони до груп.
6. Прив’язати відповідальних до груп.
7. Прив’язати або відфільтрувати події по групі.
8. Обчислити агрегований статус об’єкта з набору груп.
9. Віддати у UI:
   - список об’єктів;
   - список груп для конкретного об’єкта;
   - групові деталі для вкладок `Статус`, `Зони`, `Відповідальні`, `Події`.

#### Технічна реалізація для CASL

##### Джерела даних CASL

Для CASL вже наявні або очікувані такі джерела:

- `read_grd_object`
  - базова інформація про об’єкт;
  - список приміщень (`rooms`);
  - менеджер;
  - список `in_charge`;
- `read_connections`
  - fallback/альтернативне джерело об’єктів;
  - часто повертає об’єкт, пристрій і користувачів в одному payload;
- `read_device`
  - список ліній/зон пристрою;
  - метадані приладу;
- `read_device_state`
  - поточні стани;
  - групові стани (`state.groups`);
  - лінійні стани (`state.lines`);
- `read_user`
  - повні дані користувачів, якщо їх недостатньо в `read_grd_object` / `read_connections`;
- `read_events` / journal
  - журнал подій;
  - події, які мають бути маплені на об’єкт і групу.

##### Що вже є в коді

У поточному коді вже є корисні заготовки:

- завантаження об’єктів із `read_grd_object` / `read_connections`;
- завантаження пристроїв із `read_device`;
- завантаження стану з `read_device_state`;
- побудова `obj.Groups = mapCASLDeviceGroupsToObjectGroups(state.Groups, record.Rooms)`;
- побудова контактів через `Manager`, `InCharge`, `room.Users`.

Але зараз:

- `GetZones()` повертає зони на рівні всього об’єкта;
- `GetEmployees()` повертає контакти на рівні всього об’єкта;
- група ще не є первинною сутністю для вкладок UI.

##### Як збирати CASL-групи

Для кожного CASL-об’єкта потрібно:

1. Отримати `caslGrdObject` із `read_grd_object` або `read_connections`.
2. Отримати `caslDevice` через `read_device`.
3. Отримати `caslDeviceState` через `read_device_state`.
4. Із `state.Groups` побудувати сирий список груп.
5. Із `record.Rooms` побудувати довідник приміщень:
   - `room_id -> room name`
   - `room_id -> room users`
6. Для кожної групи з `state.Groups` створити `ObjectGroupAggregate`.
7. У групу додати:
   - `GroupNo`
   - `Status`
   - `Armed`
   - `StateText`
   - `PremiseID`
   - `PremiseName`

##### Як прив’язувати відповідальних у CASL

Оскільки в CASL відповідальні формально прив’язані до приміщення, а не до групи, backend повинен робити резолвинг у два кроки:

1. Визначити, до якого `room_id` прив’язана група.
2. Взяти всіх користувачів цього приміщення як відповідальних групи.

Порядок формування списку:

- спочатку користувачі приміщення;
- далі `in_charge`, якщо вони не дублюються;
- далі `manager`, якщо він має бути видимим у UI.

Таким чином група в CASL на рівні застосунку отримує власний список `Responsibles`, хоча фізично вони зберігаються на рівні приміщення.

##### Як прив’язувати зони у CASL

Backend повинен мати явне правило прив’язки лінії/зони до групи:

- якщо CASL API повертає явний `group_id` або `group_no` для зони, використовувати його напряму;
- якщо зона прив’язана лише до приміщення, зону треба віднести до всіх груп цього приміщення або до однієї основної групи за погодженим правилом;
- якщо немає жодної явної прив’язки, зона відноситься до першої групи об’єкта за замовчуванням.

Оскільки ти уточнив, що в оригінальному CASL кожна зона належить певній групі, а за замовчуванням першій, у ТЗ потрібно вважати це основним правилом композиції.

Практично це означає:

- потрібен проміжний резолвер `Zone -> Group`;
- у нормалізованій моделі кожна зона повинна мати рівно один `GroupID`.

##### Як формувати назву групи в CASL

Назва групи в CASL повинна формуватись окремо від назви приміщення.

Порядок пріоритету:

1. власна назва групи з CASL, якщо вона є;
2. fallback на назву приміщення;
3. fallback `Група N`.

При цьому `PremiseName` потрібно зберігати окремим полем і відображати як додатковий контекст.

##### Як прив’язувати події у CASL

Подія CASL має проходити нормалізацію в два кроки:

1. Резолв об’єкта:
   - за `obj_id`
   - або за `device_id`
   - або за `ppk_num`
2. Резолв групи:
   - за явним `group_id` / `group_no` у події;
   - або через `zone -> group`;
   - або fallback на “основну” групу об’єкта.

Події без group-контексту не повинні губитись:

- вони відображаються на рівні об’єкта;
- але не повинні ламати вкладку групових подій.

##### Що має повертати CASL provider

Для CASL provider повинен уміти віддавати:

- список об’єктів;
- список груп об’єкта;
- статус конкретної групи;
- зони конкретної групи;
- відповідальних конкретної групи;
- події конкретної групи;
- fallback-дані на рівні всього об’єкта.

#### Технічна реалізація для Phoenix

##### Джерела даних Phoenix

Для Phoenix групова модель природно випливає зі схеми SQL Server:

- `Panel`
  - кореневий об’єкт;
- `Groups`
  - основне джерело груп;
- `Company`
  - атрибути об’єкта;
- `Zones`
  - зони груп;
- `Responsibles`
  - зв’язок групи з відповідальними;
- `ResponsiblesList`
  - базові дані відповідального;
- `ResponsibleTel`
  - телефони;
- `ResponsibleEmail`
  - email;
- `vwArchives`
  - журнал подій;
- `Temp` / `CurrentAlarms`
  - активні стани/тривоги;
- `MPhone`, `OnBoardDevice`, `Channel`, `Sim`
  - технічний стан групи/об’єкта за потреби.

##### Як збирати Phoenix-групи

Для кожного `Panel_id` потрібно:

1. Завантажити root object із `Panel`/`vwRealPanel`.
2. Завантажити всі рядки `Groups` для цього `Panel_id`.
3. Для кожного рядка `Groups` створити `ObjectGroupAggregate`.

Поле групи береться з:

- `Panel_id`
- `Group_`
- `Message`
- `IsOpen`
- `TimeEvent`
- `disabled`
- іншого погодженого групового статусу.

##### Як прив’язувати зони у Phoenix

У Phoenix це найпростіше:

- `Zones.Panel_id = Groups.Panel_id`
- `Zones.Group_ = Groups.Group_`

Отже кожна зона однозначно належить конкретній групі.

Backend повинен:

1. Завантажити `Zones` для `Panel_id`.
2. Згрупувати їх по `Group_`.
3. Покласти в `ObjectGroupAggregate.Zones`.

##### Як прив’язувати відповідальних у Phoenix

Так само напряму:

- `Responsibles.Panel_id`
- `Responsibles.Group_`
- `Responsibles.ResponsiblesList_id`

Далі через:

- `ResponsiblesList`
- `ResponsibleTel`
- `ResponsibleEmail`

Backend повинен:

1. Завантажити всі `Responsibles` по `Panel_id`.
2. Для кожного запису знайти дані в `ResponsiblesList`.
3. Дотягнути телефони й email.
4. Зібрати normalized contact model.
5. Розкласти контакти по `Group_`.

##### Як прив’язувати події у Phoenix

Для Phoenix події повинні мати group-aware фільтрацію без евристики:

- `vwArchives.Panel_id`
- `vwArchives.Group_`

Отже:

1. Глобальний журнал може працювати по всіх подіях.
2. Журнал об’єкта фільтрується по `Panel_id`.
3. Журнал групи фільтрується по `Panel_id + Group_`.

Це ж саме правило має бути використане для активних тривог.

##### Що має повертати Phoenix provider

Для Phoenix provider повинен уміти віддавати:

- список об’єктів;
- список груп об’єкта;
- статус конкретної групи;
- зони конкретної групи;
- відповідальних конкретної групи;
- події конкретної групи;
- активні тривоги конкретної групи або об’єкта.

#### Як це компонувати в UI

##### Рівень 1. Список об’єктів

У списку відображається `ObjectAggregate`.

Для кожного об’єкта потрібно показати:

- назву;
- адресу;
- агрегований статус;
- кількість груп;
- індикатори джерела (`MOST`, `Phoenix`, `CASL`).

##### Рівень 2. Вибір групи

Після відкриття об’єкта UI повинен:

1. отримати список `ObjectGroupAggregate`;
2. показати список груп на вкладці `Статус`;
3. показати згруповані секції на вкладках `Зони` і `Відповідальні`;
4. не вимагати окремого глобального перемикача груп, якщо вкладки вже візуально згруповані;
5. запам’ятовувати порядок груп і стабільно відображати їх між оновленнями.

##### Рівень 3. Групові вкладки об’єкта

У group-aware режимі показувати:

- `Статус`
  - список усіх груп об’єкта;
  - для кожної групи: номер, назва, стан;
  - для `CASL` назва групи береться з назви приміщення;
  - для `Phoenix` назва групи береться з БД (`Groups.Message`);
- `Зони`
  - секції по групах;
  - у заголовку секції: номер групи, назва групи, стан групи;
  - всередині секції: номер зони, назва зони, тип зони, стан зони;
- `Відповідальні`
  - секції по групах;
  - у заголовку секції: номер групи, назва групи, стан групи;
  - всередині секції: відповідальні тільки цієї групи;
- `Події`
  - лише події цієї групи;
- `Додатково`
  - source-specific поля.

Рекомендований спосіб компонування:

- для grouped-вкладок `Зони` і `Відповідальні` використовувати `Accordion` або еквівалентні секції;
- `Tree` використовувати тільки якщо з’явиться додатковий рівень вкладеності, який реально не вкладається в `Accordion`;
- для single-group об’єктів (`MOST/міст`) зберігати поточний плоский вигляд без зайвої вкладеності.

#### Які зміни потрібні в контрактах і моделях

Потрібно еволюційно розширити моделі:

- `models.ObjectGroup` зараз занадто вузька;
- `models.Zone` не має group reference;
- `models.Contact` не має group context;
- `models.Event` не має group reference.

Мінімально потрібно додати:

- до `ObjectGroup`
  - stable string ID;
  - source;
  - group name;
  - premise metadata для CASL;
  - collections of zones/responsibles або посилання на них;
- до `Zone`
  - `GroupID`;
- до `Contact`
  - `GroupID`;
- до `Event`
  - `GroupID`.

Також потрібно додати group-aware контракти в `pkg/contracts`.

#### Рекомендований принцип реалізації

Реалізовувати це потрібно не через переписування всіх старих методів, а через паралельний group-aware шар:

- старі object-level методи залишити для сумісності;
- нові group-aware методи додати окремо;
- UI поступово переводити на групову модель;
- multi-source aggregator повинен маршрутизувати запити на рівні `ObjectID` і `GroupID`.

Додаткові правила реалізації:

- source-specific логіка не повинна потрапляти у Fyne-віджети;
- правила виду “для `CASL` назва групи = назва приміщення” і “для `Phoenix` назва групи = `Groups.Message`” повинні жити у provider/mapper шарі;
- UI повинен працювати з уже нормалізованими доменними моделями;
- реалізацію слід будувати через MVVM:
  - models/data provider -> доменні сутності;
  - viewmodel -> групування, сортування, presentation formatting;
  - ui -> лише bind/render/layout;
- код повинен залишатись чистим:
  - без змішування SQL-логіки і UI;
  - без `if source == ...` усередині Fyne-компонентів;
  - без дублювання форматування між вкладками.

#### Підсумкове технічне правило

У `Phoenix` група є фізичною сутністю БД.

У `CASL` група є логічною оперативною сутністю, яка:

- має власний статус;
- має власні зони;
- має власних відповідальних через приміщення;
- повинна бути головною сутністю робочої області застосунку після вибору об’єкта.

## 7. Вимоги до реалізації по шарах

## 7.1. Конфігурація

Потрібно розширити `DBConfig`:

- додати режим `phoenix`;
- додати параметри SQL Server:
  - `server`;
  - `instance`;
  - `database`;
  - `user`;
  - `password`;
  - `encrypt`;
  - `trust_server_certificate`;
  - `integrated_security` або еквівалент;
  - додаткові параметри.

Поточне поле `Path` для Phoenix не підходить.

Окремо потрібно передбачити, що Phoenix-конфігурація не обов’язково заміняє Firebird-конфігурацію, а може зберігатися паралельно з іншими джерелами.

Мінімально потрібно підтримати:

- окремий блок/секцію налаштувань Phoenix;
- ознаку `enabled` для Phoenix;
- пріоритет або правила маршрутизації Phoenix у multi-source режимі;
- можливість увімкнути Phoenix без вимкнення інших джерел.

### 7.2. UI налаштувань

Діалог налаштувань БД має:

- мати окрему вкладку налаштувань для Phoenix;
- не змішувати Phoenix-поля з Firebird/MOST-полями в одному блоці;
- дозволяти окремо вмикати/вимикати Phoenix як джерело;
- показувати різні поля для Firebird і Phoenix;
- не вимагати шлях `.FDB`, якщо налаштовується Phoenix;
- підтримувати перепідключення/оновлення конфігурації Phoenix без втрати інших підключених джерел.

Рекомендована структура вкладок:

- `MOST / Firebird`
- `Phoenix`
- `API / CASL / інші джерела`
- `Загальні правила multi-source / маршрутизації`

### 7.3. Підключення до БД

Потрібно:

- додати драйвер SQL Server;
- винести в окрему абстракцію параметри драйвера, DSN і назву СУБД;
- уніфікувати `InitDB(...)`, щоб вона не була прив’язана до Firebird;
- переглянути health-check і логування, щоб вони були нейтральними до типу БД.

### 7.4. Провайдери

Потрібно створити окремий Phoenix-провайдер.

Вимога:

- Firebird-код не повинен обростати умовами `if mode == phoenix` всередині кожного запиту;
- Phoenix має бути окремою реалізацією контракту.
- Phoenix admin має бути окремим провайдером/модулем, а не адаптацією поточного Firebird admin через міст.
- Phoenix повинен підключатися до агрегатора джерел як окремий source, а не як глобальна заміна решти backend-джерел.

### 7.4.1. Робота в multi-source режимі

У Phoenix-інтеграції потрібно передбачити:

- підключення Phoenix одночасно з іншими БД;
- підключення Phoenix одночасно з API-джерелами;
- правила, які об’єкти/події/тривоги належать Phoenix;
- поведінку UI, коли однакові або схожі сутності приходять з кількох джерел;
- окремі ідентифікатори або namespace-правила, щоб уникнути конфліктів між `OBJN`, `Panel_id`, CASL ID та іншими ключами.

### 7.5. SQL та діалект

Потрібно врахувати:

- `TOP 1` замість `FIRST 1`;
- відмінності у placeholder-ах;
- `WITH (NOLOCK)` як окрема свідома опція, а не автоматичне копіювання у всі запити;
- відмінності типів `bit`, `datetime`, `nvarchar`, `varchar`;
- інший синтаксис конвертацій, сортувань, `COALESCE/ISNULL`.

### 7.6. Мапінг доменних моделей

Потрібно описати й протестувати мапінг:

- Phoenix status/open state -> `models.Object.Status`, `StatusText`;
- Phoenix event code/type -> `models.Event.Type`;
- Phoenix zone status -> `models.Zone.Status`;
- Phoenix flags `Disabled`, `TestPanel`, `IsOpen`, `Panel_Type` -> внутрішні прапорці UI.

## 8. Окреме рішення по admin-функціоналу

### 8.1. Поточний стан

Admin у проєкті дуже глибоко зав’язаний на Firebird-схему:

- створення об’єктів;
- оновлення об’єктів;
- видалення;
- зони;
- персонал;
- координати;
- довідники;
- emulation;
- display blocking;
- monitoring settings;
- message override;
- subserver bindings;
- статистика та перевірки цілісності.

### 8.2. Базова вимога для Phoenix

Для Phoenix admin-функціонал має бути іншим за логікою і не залежати від мосту.

Це означає:

- не можна проєктувати Phoenix admin як “ще один режим” поверх існуючих Firebird admin-запитів;
- не можна вимагати наявності MOST/Firebird БД або мосту для роботи Phoenix admin;
- не можна використовувати Firebird mirror-таблиці, якщо Phoenix має власну первинну модель;
- потрібно окремо описати Phoenix-native сценарії створення, редагування, блокування, роботи з зонами, відповідальними, каналами та пристроями;
- якщо певна дія доступна лише через Phoenix API/процедури/службові механізми, саме вони мають вважатись основним шляхом, а не обхід через міст.

### 8.3. Висновок по обсягу

Повна підтримка Phoenix для admin-розділу є окремим великим підпроєктом.

Рекомендовано поділити роботи на дві черги:

### Черга 1

- тільки read-only сценарії;
- без запису в Phoenix;
- без admin-меню або з частковим відключенням недоступних пунктів.

### Черга 2

- окремий Phoenix Admin Provider;
- окремий Phoenix-native admin UX/процеси там, де предметна модель відрізняється від Firebird;
- окремо погоджені бізнес-процеси створення/редагування об’єкта;
- окремо погоджені SQL-операції запису;
- окремо погоджені правила роботи з групами, зонами, відповідальними, каналами та приладами.

### 8.4. Мінімальний admin-scope, який можна розглядати після read-only

- перегляд картки об’єкта;
- перегляд зон;
- перегляд відповідальних;
- перегляд каналів;
- перегляд параметрів пристрою.

Редагування дозволяти тільки після підтвердження бізнес-правил і таблиць запису.

## 9. Етапи впровадження

### Етап 1. Архітектурна підготовка

- розширити конфігурацію режимом `phoenix`;
- додати універсальну ініціалізацію БД;
- додати фабричний вибір провайдера;
- інтегрувати Phoenix у multi-source агрегатор;
- оновити діалог налаштувань і додати окрему вкладку Phoenix;
- зберегти зворотну сумісність з Firebird.

### Етап 2. Read-only Phoenix Provider

- список об’єктів;
- картка об’єкта;
- зони;
- відповідальні;
- журнал подій по об’єкту;
- глобальний журнал подій;
- базова технічна інформація по приладу/каналу.

### Етап 3. Панель активних тривог

- погодити джерело активних тривог;
- реалізувати Phoenix-еквівалент `GetAlarms()`;
- перевірити відповідність поточному UX.

### Етап 4. Тестові/технічні вкладки

- підтягнути останні тести;
- підтягнути канал і SIM;
- окремо вирішити, чи є в Phoenix історія тестових повідомлень.

### Етап 5. Admin Phase 2

- погодити Phoenix-native write-model без залежності від мосту;
- реалізувати тільки ті сценарії, для яких є підтверджені таблиці, FK, обмеження й бізнес-правила саме в Phoenix;
- додати окремі інтеграційні тести на транзакції.

## 10. Нефункціональні вимоги

- існуючий Firebird-режим не повинен деградувати;
- Phoenix-режим повинен коректно перепідключатись через налаштування;
- помилки підключення мають бути зрозумілими в UI та логах;
- SQL-запити мають бути параметризованими;
- усі операції БД мають працювати через `context`;
- треба уникати копіювання Firebird-логіки всередину Phoenix-провайдера без переосмислення моделі;
- потрібні unit-тести мапінгу;
- потрібні інтеграційні тести, якщо є доступ до тестового Phoenix-середовища.

## 11. Критерії приймання

Робота вважається прийнятою для першої черги, якщо:

- застосунок підтримує одночасно `MOST/міст`, `Phoenix` і `CASL API` в межах єдиної multi-source конфігурації;
- у налаштуваннях є окрема вкладка Phoenix;
- Phoenix можна увімкнути як окреме джерело даних;
- застосунок успішно підключається до Phoenix;
- Phoenix може працювати одночасно з іншими БД та/або API-джерелами;
- список об’єктів відкривається і відображає коректні записи;
- картка об’єкта показує базові дані;
- зони відображаються коректно;
- відповідальні відображаються коректно;
- журнал подій працює;
- глобальний event loop не падає;
- оновлення конфігурації Phoenix працює без перезапуску застосунку;
- у Phoenix-режимі недоступні сценарії не падають, а коректно вимикаються або показують зрозуміле повідомлення.

## 12. Ризики

### 12.1. Концептуальні

- у Firebird об’єкт фактично сприймається як одна сутність, у Phoenix є виражена модель `Panel + Group`;
- частина UI поточно спирається на число `OBJN`, а в Phoenix ідентифікатор схожий на рядковий `Panel_id`;
- модель відповідальних у Phoenix багатша, ніж поточна модель `Contact`.

### 12.2. Технічні

- відсутній підтверджений SQL для активних тривог;
- відсутній підтверджений SQL для історії тестових повідомлень;
- невідомо, які view гарантовано є в усіх інсталяціях Phoenix;
- невідомо, чи дозволене `NOLOCK` і чи це взагалі бажана поведінка;
- admin-операції можуть вимагати запис не в 1-2 таблиці, а в пов’язаний набір сутностей з жорстким порядком.

## 13. Якої інформації не вистачає

Нижче перелік інформації, без якої реалізація буде або ризиковою, або неповною.

### 13.1. По підключенню

- у документації вже є базові параметри підключення:
  - сервер/інстанс: `localhost\PHOENIX4`;
  - база: `Pult4DB`;
  - логін: `sa`;
  - пароль: порожній;
- але все ще не вистачає підтвердження, який саме драйвер/рядок підключення вважається цільовим для Go (`sqlserver`, `odbc`, інший);
- чи достатньо тільки SQL-auth, чи треба підтримати ще й Windows-auth;
- чи потрібні `encrypt`, `trustServerCertificate`, таймаути, named instance discovery;
- чи база завжди `Pult4DB`, чи це має бути параметр.

### 13.1.1. По multi-source конфігурації

- як саме Phoenix має співіснувати з іншими джерелами: primary/secondary/equal peer;
- які правила ownership для Phoenix у змішаному режимі;
- чи потрібно дозволяти одночасне увімкнення Phoenix + Firebird + CASL;
- як розв’язувати конфлікти однакових об’єктів між різними джерелами;
- чи потрібен окремий префікс/namespace для ID Phoenix в UI та внутрішній маршрутизації.

### 13.2. По ідентифікації об’єкта

- що вважати “ID об’єкта” у UI для Phoenix:
  - тільки `Panel_id`;
  - `Panel_id + Group_`;
  - внутрішній surrogate key;
- чи група `0` завжди є основною;
- як поводитись з об’єктами без групи `0`.

### 13.3. По активних тривогах

- точний затверджений SQL-запит для списку активних тривог;
- що саме в Phoenix вважати “активною тривогою” для екрану `AlarmPanel`;
- як відрізняти пожежні, охоронні, технічні й службові тривоги.

### 13.4. По журналу подій

- чи можна стабільно використовувати `vwArchives.Event_id` як монотонний курсор;
- чи є обмеження по продуктивності `vwArchives` на великих обсягах;
- який рекомендований часовий фільтр для первинного завантаження.

### 13.5. По технічній інформації

- що є офіційним аналогом Firebird-полів:
  - `TBL_TESTCONTROL`;
  - `TRPMSG`;
  - зовнішніх підсерверних БД;
- де брати історію тестових повідомлень, якщо вона взагалі існує в Phoenix.

### 13.6. По admin-операціях

- офіційний сценарій створення нового об’єкта в Phoenix із повним переліком таблиць;
- які поля обов’язкові для створення охоронного об’єкта;
- які поля обов’язкові для створення пожежного об’єкта;
- порядок створення зв’язаних записів;
- допустимі значення для довідників;
- які обмеження й тригери є в БД;
- які операції заборонено виконувати напряму;
- чи є службові процедури/API, через які треба робити запис замість прямого SQL;
- чи існують Phoenix-native admin-сценарії, які принципово не мають аналога в MOST;
- які саме admin-функції Phoenix повинні бути незалежними від мосту вже в першій повній версії admin-модуля.

### 13.7. По довідниках і статусах

- затверджена мапа:
  - `IsOpen`;
  - `Disabled`;
  - `TestPanel`;
  - `Panel_Type`;
  - `Zones.Status`;
  - `TypeCode.idTCode`;
- перелік кодів подій, які потрібно показувати як `Fire`, `Fault`, `Restore`, `Arm`, `Disarm`, `Offline`.

### 13.8. По середовищу тестування

- доступ до тестової Phoenix-бази;
- облікові дані з правами лише на читання для першої черги;
- окремий стенд для запису, якщо буде admin-фаза;
- 5-10 реальних контрольних об’єктів для ручної верифікації.

Рекомендую формально затвердити такий обсяг:

### Спринт 1

- окрема вкладка налаштувань Phoenix;

## 15. Підсумок

Підтримка Phoenix для цього проєкту технічно можлива, але це не “переключення драйвера”, а окремий адаптер під іншу предметну модель.

Найбільш реалістичний і безпечний шлях:

- зберегти одночасну підтримку `MOST/міст`, `Phoenix` та `CASL API`;
- спочатку реалізувати Phoenix як read-only джерело даних;
- відразу інтегрувати Phoenix як окреме джерело в multi-source модель, а не як заміну інших БД/API;
- не змішувати Firebird і Phoenix SQL в одному провайдері;
- admin-функції Phoenix винести в окрему другу чергу й реалізовувати їх як незалежний від мосту контур;
- до початку реалізації отримати підтвердження по активних тривогах, ідентифікації об’єкта та запису в Phoenix.
